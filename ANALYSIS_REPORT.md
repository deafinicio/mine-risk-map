# Аналіз репозиторію `mine-risk-map`

## 1) Структура та призначення файлів
- `index.html` — основний вебдодаток на Leaflet: карта, шари, фільтри, легенда, завантаження GeoJSON.
- `data/sessions.geojson` — точкові дані сесій навчання (дата, громада, аудиторія, кількість учасників).
- `data/admin_agg.geojson` — полігони громад із агрегованим полем `sum_participants`.
- `script_test.js` — окремий JS-файл із логікою, що майже дублює вбудований скрипт з `index.html` (ймовірно тестова/альтернативна версія).
- `README.md` — коротка інструкція запуску через GitHub Pages.

## 2) Функціональний огляд

### Карта та базові шари
Реалізовано 3 тайлові шари (OSM, Carto, OSM.de) з каскадним fallback при `tileerror`, що підвищує надійність відображення. Центр карти — Харківська область.

### Хороплет громад
Полігони з `admin_agg.geojson` фарбуються за порогами `sum_participants`, є popup, hover-виділення та легенда. Для демо-даних підхід коректний.

### Точкові сесії
Для `sessions.geojson` реалізовано:
- кластеризацію маркерів;
- часовий шар (`TimeDimension`) за наявності бібліотеки;
- heatmap із нормалізацією інтенсивності по `participants`.

### Пошук і фільтрація
Є фільтр за текстом (громада/район/аудиторія) та діапазоном дат, а також кнопка «Оновити дані» із cache-busting (`?nocache=<timestamp>`).

## 3) Перевірка коректності

### Що працює коректно
1. GeoJSON-файли валідні як `FeatureCollection` і містять очікувані типи геометрій (Point/Polygon).
2. Код стійко обробляє відсутні поля через дефолтні значення (`||`, `??`, перевірки `!= null`).
3. Наявні `catch(console.error)` для основних `fetch`-операцій.

### Потенційні проблеми/ризики
1. **Дублювання логіки**: `script_test.js` майже повторює `index.html`. Це ускладнює підтримку й створює ризик розсинхронізації.
2. **Фільтр кінцевої дати**: `end-date` парситься як дата на `00:00:00`, тому записи в той самий день, але пізніше за північ, можуть помилково виключатися.
3. **Немає захисту від невалідного `datetime`** у частині порівняння дат (можливий `Invalid Date` при «брудних» даних).
4. **Хороплет не синхронізується автоматично з точками**: `sum_participants` береться тільки з `admin_agg.geojson`, а не перераховується з `sessions.geojson`.

## 4) Рекомендації
1. Винести JS-логіку з `index.html` в один модульний файл (наприклад, `app.js`) і видалити/перепрофілювати `script_test.js`.
2. Для `end-date` застосовувати кінець доби (23:59:59.999) або порівняння тільки по даті.
3. Додати валідацію даних перед рендерингом (тип геометрії, обов'язкові поля, валідність дати).
4. За потреби зробити динамічну агрегацію `sum_participants` із `sessions.geojson` (або чітко задокументувати, що агрегат — окреме джерело).

## 5) Загальний висновок
Проєкт написаний акуратно для starter/demo-рівня: функціонал карти, фільтрів і візуалізацій реалізований робоче та читабельно. Критичних синтаксичних проблем не виявлено. Основні покращення стосуються підтримуваності (усунення дублювання) і edge-case коректності у фільтрації дат.
