# Mine Risk Map — Kharkiv Oblast

Інтерактивна геоінформаційна система для збору, нормалізації, зберігання та візуалізації даних про проведення сесій інформування населення щодо ризиків вибухонебезпечних предметів (MRE) у Харківській області.

Проєкт об’єднує польовий збір даних, централізоване зберігання та аналітичну візуалізацію на карті з можливістю фільтрації, heatmap-аналізу та експорту у звітні матеріали.

---

## 1. Мета проєкту

### Основна мета

Створити єдину систему, яка дозволяє:

- швидко збирати дані про MRE-сесії;
- стандартизувати структуру даних;
- автоматично прив’язувати події до географії;
- візуалізувати охоплення населення;
- формувати аналітику та звітність.

### Операційні цілі

- усунути ручну обробку Excel/Word довідок;
- мінімізувати помилки у геоприв’язці;
- забезпечити єдину схему даних для всіх років;
- надати інструмент для швидкого аналізу покриття;
- спростити підготовку донорських звітів.

---

## 2. Ключова ідея

Проєкт — це не просто карта.

Це **повноцінна інформаційна система MRE-напрямку**, яка працює за принципом:

> введення даних → нормалізація → централізоване зберігання → геоприв’язка → візуалізація → аналітика → експорт

---

## 3. Архітектура рішення

### Загальна схема

```text
Web Form
   ↓
Google Apps Script (doPost)
   ↓
Google Sheets (Database)
   ↓
Google Apps Script (doGet)
   ↓
Frontend (Leaflet Map)
   ↓
Visualization (Markers + Clusters + Heatmap)
```

---

## 4. Потік даних (Data Flow)

### 4.1. Введення даних

1. Інструктор або координатор відкриває веб-форму
2. Заповнює:
   - район
   - громаду
   - населений пункт
   - дату
   - інструкторів
   - кількість бенефіціарів
3. Дані відправляються POST-запитом

### 4.2. Запис у базу

- Google Apps Script (doPost) приймає дані
- перевіряє структуру
- додає новий рядок у Google Sheets

### 4.3. Отримання даних

- карта викликає doGet()
- отримує JSON масив

### 4.4. Геоприв’язка

Frontend:

- звіряє `raion + hromada + settlement`
- знаходить відповідну точку у GeoJSON

### 4.5. Візуалізація

- створюються маркери
- формуються кластери
- будується heatmap

## 7. Логіка роботи карти

### 7.1. Маркери

Кожен запис = одна сесія.

- відображається як точка
- має popup з деталями

### 7.2. Кластери

- групують точки при zoom-out
- розкриваються при zoom-in

### 7.3. Heatmap

Будується за `participants_total`

Принцип:

- більше людей → сильніша інтенсивність
- менше людей → слабка інтенсивність

---

## 8. Логіка даних

### Один запис = одна сесія

Навіть якщо:

- одна дата
- одна локація
- ті самі інструктори

— це різні сесії.

### Основні поля

- created_at
- raion
- hromada
- settlement
- session_date
- instructor_1
- instructor_2
- participants_total
- participants_u18
- participants_18plus
- notes

---

## 9. Нормалізація даних

### Принципи

- settlement = тільки назва
- адреси → в notes
- назви узгоджені з довідником

### Приклади

```
м. Лозова, 4 мкрн → Лозова
Харків, вул. Шевченка → Харків
```
